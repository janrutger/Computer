DEF main {

     2. Setup Sockets (Inbox Deques)
    # destination on Port 100
    DEQUE.new AS out-msgbox
    100 out-msgbox SOCKET.bind
    "Bound Port 100 (Server)\n" PRTstring

    # Client on Port 101
    DEQUE.new AS in-msgbox
    101 in-msgbox SOCKET.bind
    "Bound Port 101 (Client)\n" PRTstring


    RDP.init IF {                                                   # Verwacht geen argumenten
        RDP.connect AS connection                                   # Verwacht \"dest_id" dest_port local_port > conn_id | 0
        connection IF {
            1 AS can_send
            WHILE true DO
            
                in-msgbox   connection RDP.update                   # verwacht SDP mailbox pointer en en connection pointer
                out-msgbox  connection RDP.update                   # verwacht SDP mailbox pointer en en connection pointer
        
        
                app-buffer connection RDP.receive IF {              # kopieert data naar app-buffer, geeft internal buffer vrij
                    application handles IN-buffer
                }
        
                can_send IF {
                    Application creates new OUT-buffer to send
                    2 AS can_send
                }
        
                can_send IF {
                    2 can_send == IF {
                        OUT-buffer connection RDP.send IF {         # verwacht een OUT-buffer en een connection pointer        #
                            1 AS can_send
                        }
                    }
                }
            DONE
        }
    }
}

# ==============================================================================
# SRDP (Simple Reliable Data Protocol) - ARCHITECTUUR & SPECIFICATIE
# ==============================================================================
# "Keep it Simple, Make it Fun"
# Filosofie: Dit is een protocol voor het Stern-ATX ecosysteem. 
# ==============================================================================

# --- STERN-ATX GEHEUGEN CONSTRAINTS (CRITIEK) ---
# 1. GEHEUGEN LAYOUT: De Stern-ATX beschikt over 24KB (words) systeemgeheugen.
#    Daarnaast is er een specifieke HEAP van 6KB (6144 words) voor objecten.
# 2. LINEAIRE HEAP: De 'std_heap' allocator beweegt de HEAP_FREE pointer alleen
#    vooruit. Er is GEEN Garbage Collector. Objecten kunnen NOOIT echt worden
#    verwijderd of 'vrijgegeven' (geen free() functie).
# 3. SIZE AGNOSTIC EMULATIE: Hoewel de Python emulator (memoryR3x.py) flexibel
#    omgaat met integer groottes, moet de Stacks-code ontworpen worden voor
#    16-bit precisie en strikte geheugenlimieten.
# 4. NO-NEW-IN-LOOP: Het aanmaken van heap-objecten (NEW.list, DICT.new) binnen
#    de main-loop of RDP.update is VERBODEN. Dit leidt onherroepelijk tot een
#    Heap Overflow omdat de vrije ruimte op de heap nooit toeneemt.

# --- BUFFER POOL & DICTIONARY STRATEGIE ---
# 1. BUFFER POOL: We pre-alloceren 16 buffers van 64 words via RDP.init.
#    - Pakketten worden 'geleend' uit de pool (via een Deque) en MOETEN worden 
#      teruggegeven via RDP.ret_buf zodra ze bevestigd of verwerkt zijn.
# 2. DICTIONARY GEBRUIK: TX_BUF en RX_BUF mappen SeqNum (Key) -> BufferPointer (Val).
#    - De Dictionary beheert intern zijn slots. Door DICT.remove te gebruiken 
#      na een ACK, blijft de Dictionary compact binnen zijn pre-allocated capaciteit.

# --- STATE HIËRARCHIE ---
# | Variabele        | Toestand      | Betekenis                                  |
# |------------------|---------------|--------------------------------------------|
# | CONN_STATE       | CLOSED (0)    | Geen actieve sessie.                       |
# | (De Levenscyclus)| LISTEN (1)    | Wacht op inkomende HELLO.                  |
# |                  | INIT (2)      | Bezig met handshake (verzend HELLO).       |
# |                  | ACTIVE (3)    | Verbinding staat, data-overdracht mogelijk.|
# |------------------|---------------|--------------------------------------------|
# | CONN_STATE_OUT   | READY (0)     | Mag direct DATA sturen.                    |
# | (Zend Flow)      | WAIT (1)      | Gepauzeerd door peer (ontvangen WAIT).     |
# |                  | REQ (2)       | Wacht op STAT_ACK na een venster-burst.    |
# |------------------|---------------|--------------------------------------------|
# | CONN_STATE_IN    | READY (0)     | Kan data ontvangen.                        |
# | (Ontvang Flow)   | WAIT      (1) | Buffer raakt vol (>8), WAIT is verzonden.  |
# |                  | RESUME        | er is weer ruimte, Resume in verzonden     |
# |                  | TIME_WAIT     | Wacht na BYE op laatste pakketten.         |
# |------------------|---------------|--------------------------------------------|

# --- ARCHITECTURAL PRINCIPLE: EXCLUSIVE PORT OWNERSHIP ---
# Om "Double Read" conflicten te voorkomen, geldt de volgende strikte regel:
# 1. Zodra een poort (Deque/msgbox) aan een RDP-verbinding is gekoppeld, 
#    mag de applicatie NOOIT meer direct SOCKET.read aanroepen op deze poort.
# 2. RDP.update is de EXCLUSIEVE consument van de msgbox_ptr.
# 3. Alle inkomende data (zowel RDP-protocol als applicatie-data) wordt door 
#    RDP.update uit de Deque gehaald en gesorteerd.
#
# --- SEQUENCE WRAP-AROUND LOGICA ---
# 1. 16-BIT BEREIK: Volgnummers lopen van 1 t/m 65535 (startwaarde is 1).
# 2. CHECK: (NewSeq - LastSeq) & 0xFFFF. 
#    Resultaat < 32768 is NIEUW, > 32768 is OUD/DUPLICAAT.
#
# --- Functie Definities ---
# RDP.update ( msgbox_ptr connection_ptr -- )
# 1. Inbox Legen:
#    - Haal bericht uit msgbox_ptr.
#    - IS HET EEN CONTROLE PAKKET (Type 131-137)?
#        * ALTIJD VERWERKEN. Deze nemen nauwelijks ruimte in en zijn 
#          nodig om de verbinding open te houden of te herstellen.
#    - IS HET EEN DATA PAKKET (Type 130)?
#        * CHECK RX_BUF DICT.count. 
#        * IS COUNT < RDP_RX_WINDOW_SIZE (bijv. 5)?
#            - Accepteer: Plaats in RX_BUF, update SEQ_IN.
#        * IS COUNT >= RDP_RX_WINDOW_SIZE?
#            - Weiger: Laat het pakket in de msgbox_ptr staan (STOP met legen)
#              OF (beter) gooi het weg en stuur een STAT_ACK/WAIT. 
#              De zender zal het later herverzenden wanneer er weer ruimte is.
#
# 2. Backpressure Signalering:
#    - Als de RX_BUF vol raakt (count == limit):
#        * Zet status op WAIT en stuur RDP_MSG_WAIT naar de zender.
#    - Als de applicatie data verbruikt via RDP.recv en de count zakt onder de grens:
#        * RDP.recv moet dan een signaal (RESUME) triggeren.
#
# RDP.recv ( app_buffer_ptr connection_ptr -- success )
# - De applicatie gebruikt UITSLUITEND deze functie om data te ontvangen.
# - Werking:
#     1. Kijk in CONN_RX_BUF of de key 'CONN_SEQ_IN' aanwezig is.
#     2. Indien aanwezig: kopieer payload naar app_buffer_ptr, verwijder uit 
#        dictionary, hoog CONN_SEQ_IN op en geef success (1).
#     3. Indien niet aanwezig: geef 0 (geen data beschikbaar of gat in sequentie).

# --- RDP API Functies ---

# RDP.init ( -- success )
# - Initialiseert de Packet Pool en pre-alloceert de 16 buffers op de 6K heap.

# RDP.get_buf ( -- buffer_ptr | 0 )
# - Haalt een vrije buffer uit de pool. Geeft 0 als de pool leeg is.

# RDP.ret_buf ( buffer_ptr -- )
# - Geeft een buffer terug aan de pool voor hergebruik.

# RDP.update ( msgbox_ptr connection_ptr -- )
# - Verwerkt inkomende pakketten en voert de state-machine logica uit.
# - Handelt wrap-around, idle-timeouts, flow-control en MAX_RETRIES af.

# RDP.send ( app_buffer_ptr connection_ptr -- success )
# - Verstuurt data als de status ACTIVE/READY is. 
# - Kopieert data naar pool-buffer en plaatst in TX_BUF.

# RDP.flush ( connection_ptr -- )
# - Forceert een STAT_REQ als er onbevestigde data in de TX_BUF zit.

# RDP.receive ( app_buffer_ptr connection_ptr -- success )
# - Kopieert in-order data naar de app_buffer_ptr en geeft de pool-buffer direct vrij.
# - Geeft success (1) of 0 als er geen data beschikbaar is.

# ==============================================================================
# AANDACHTSPUNTEN BIJ IMPLEMENTATIE (CHECKLIST)
# ==============================================================================
# 1. DEADLOCK PREVENTIE: Gebruik de statische `_send_buffer` van socket_lib voor 
#    controle-berichten (ACK, WAIT, STAT_REQ, etc.) in plaats van de Packet Pool. 
#    Dit garandeert dat protocol-updates altijd verstuurd kunnen worden, ook 
#    als de pool volledig bezet is door DATA-pakketten.
# 2. INBOX DRAINING: Gebruik een WHILE loop in `RDP.update` om alle wachtende 
#    pakketten in de Deque in één keer te verwerken. Dit voorkomt latency en 
#    overloop van de kernel-buffers bij bursts.
# 3. DICTIONARY ITERATIE: Bij het verwijderen van items in een loop (bijv. 
#    timeout checks), houd rekening met het "swap-with-last" mechanisme van 
#    `DICT.remove`. Hoog de index teller NIET op na een verwijdering.
# 4. SEQUENCE LOGICA: Gebruik strikt de `(New - Last) & 0xFFFF` vergelijking om 
#    de 16-bit wrap-around correct af te handelen. Een resultaat < 32768 is 'nieuw'.
# 5. EXCLUSIEF EIGENAARSCHAP: De applicatie mag NOOIT `SOCKET.read` aanroepen op 
#    een poort die door SRDP wordt beheerd; dit "steelt" pakketten van de state-machine.
# 6. RX THRESHOLDS: Implementeer de "Low Water Mark" (bijv. onder 5 pakketten) 
#    voor het sturen van RESUME om onnodig "klapperen" van de flow-control te voorkomen.
# 7. FLOW CONTROL HYSTERESIS: Gebruik High/Low watermarks voor de RX_BUF. 
#    Stuur WAIT bij een volle dictionary (High), 
#    maar stuur pas RESUME wanneer de applicatie de dictionary heeft leeggegeten tot onder 
#    de drempelwaarde (low water mark).
